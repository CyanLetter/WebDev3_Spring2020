The System

Git is a version control system.

Maintains history of a project, so no work gets lost.

Can be used without a website or app.

Use. Git. For. Every. Project.

Starting a Git repository without Github:

  1. cd
    and drag your project folder into terminal.
  2. git init

That's it, you now have a (local) git repository

You can use Terminal commands from here, With commands like: 

git status
git add -A
git commit -m "Initial Commit!"

Terminal command structure



command  option  option  argument
git      commit  -m      "initial commit"

You can also just drag into Github Desktop

Important Folders

.git and .gitignore

.git folder

This is your git repository.

Includes config files and your project history.

.gitignore

Prevents certain files and folders (like node-modules) from being added to the repo.

Right click on files in Github Desktop and click 'ignore'.

Can edit this directly.

Never commit node_modules

No Repo, No Project

Or how to start completely fresh.

  1. Create a folder in Finder
  2. Drag folder onto Github Desktop
  3. Click "create repository"
  4. Add files, Publish

That's it. Start adding project files.

Alternately...

  1. Go to Github.com
  2. Create a new repository
  3. Hit 'Clone in Desktop'

Project, No Repo

Or how to create a git repository from an existing project

  1. Drag project folder into Github Desktop
  2. Hit 'Publish'

This will create a repository on github.com

Project and Repo

Or how to add an existing project to an existing repo.

(Usually you will not want to do this)

  1. Clone repository into new folder
  2. Drag files from existing project (not .git)
  3. Commit

Essentially copying your project into a repository folder.

The Basics

Cloning, committing, pushing, pulling.

Cloning

Download a LOCAL copy of a git repository from a REMOTE server, including all history.

cd path/to/your/project

git clone https://github.com/CyanLetter/WebDev3_Spring2020

Committing

Save a particular set of changes in your local copy.

  • Does not affect the remote yet.
  • Commit early, commit often.
  • Commit before the end of the day.
  • There is no such thing as too many commits.
git add [files to add]

git commit -m "Your message"

Pushing, pulling

  • PULL or FETCH any new commits that the remote server has down to your local copy
  • PUSH the local commits you made back to the remote server
  • Always pull before pushing your own commits!

About pushing

git pull

git push origin master

Keeping up-to-date

When you sync, you can see how many commits your local copy is behind or ahead of the remote.

Reverting

Restoring files to a previous version

There are about a million ways to do this.

Most of them are confusing or dangerous.

Uncommitted changes

In the app, right click on a file and select Discard Changes, or Discard All

Opening a specific file
  • Select a commit
  • Right click on a file and select "open"
  • Copy and paste as necessary

Going back to previous commits

Lots of dangerous ways to do this that rewrite history.

The command line is simplest

The Undo button

DANGER

REWRITES HISTORY

DANGER

In the history tab, right click the commit you want to undo and select 'Copy SHA'

In Terminal:

git revert --no-commit 0766c053..HEAD

Make sure to include the '..HEAD' after your SHA

The app will create a new commit for you.

Branching

Making an alternate timeline

Different workflows

The master branch should never be broken.

Branches let you work without breaking master.

Great for working in teams, still helpful if it's just you.

Stay in sync with master

Always branch from master.

Always update from master (in app).

Merging

Connecting the timelines.

When you're finished with a branch...

  • Switch to the branch you want to merge into (e.g. master)
  • Go to Branch > Merge Into Current Branch
  • Pick the branch you want to merge
  • Resolve conflicts, commit

Forking

Duplicating someone else's repo for yourself to edit.

The fork button

On github. Creates a copy of the project that you own.

You can pull updates from the original project.

Pull requests

A request to merge your changes back into the original repo.

Best to do this from a branch.

This is how people fix bugs and add features in open source projects.

Setting up a merge so that it can be reviewed.

Always make a request to merge your current branch into the target branch.

About pull requests

From Desktop

  • Make sure you're in the right branch
  • Hit the Pull Request button
  • Add a description, send it.
  • Owner can review and accept on github.

From the Web

  • Make sure you're in the right branch
  • Hit the New Pull Request button
  • Add description, this sould all be familiar
